# Reference Data Layer

🟢 `[status: active]` `[lane: data]` `[type: spec]`

## Purpose

The reference layer helps with card design and migration; it is not required for rendering.

## Sources

### MTGJSON

Use for bulk local card/set/ruling metadata.

Recommended files:

- `AllPrintings.json` or SQLite for full data,
- set-specific files when performance matters,
- CSV exports if the app needs spreadsheet-like queries.

### Scryfall

Use for:

- named card lookup,
- official wording examples,
- source-card imports,
- card image/art-crop references,
- Oracle text comparison.

Respect Scryfall API terms/rate limits. Cache results locally.

## Local cache shape

```text
reference/
  custom/
    references.json
  official/
    current/
      catalog.json
      rules.json
      update-status.json
    history/
    reports/
reference-cache/
  mtgjson/
    metadata.json
    AllPrintings.sqlite
  scryfall/
    cards/
    bulk/
    images/
```

`reference/custom/references.json` is the local project-owned store for terms
created inside Homebrew Forge. It sits beside the included catalog rather than
editing seed data. Each saved reference carries its category, source, system,
official/homebrew origin, lifecycle status, and draft/final workflow status.

`reference/official/current/` is the app-readable snapshot for reviewed
official data generated by `forge reference` commands. Source snapshots,
previous current files, and diff reports live under `reference/official/history`
and `reference/official/reports`. The editor reads these files locally; it does
not fetch Scryfall, MTGJSON, Wizards rules, or MCP data on startup.

The References workspace plus button uses the shared creation overlay shell. It
creates one term at a time or previews one pasted JSON/CSV/XML-ish reference
before saving. Bulk reference import belongs to the future File > Import hub.

The References workspace keeps ordinary search visible and sends advanced
filters through the shared browse/filter overlay. The first usage-aware slice
indexes the active set and selected card locally, so terms can be narrowed by
whether they appear in the current card pool. Deck-, collection-, and broader
project-wide usage filters should load those datasets explicitly before they are
presented as complete.

## Reference commands

```bash
forge reference sync --source mtgjson
forge reference sync --source scryfall --bulk oracle-cards
forge reference audit --source scryfall --json
forge reference sync --source scryfall --dry-run --report reference/official/reports/reference-sync
forge reference rules sync --dry-run --report reference/official/reports/rules-sync
forge reference update --check
forge reference search "o:investigate color<=uw"
forge reference named "Lightning Bolt"
forge compare --card DEMO-004 --against "Lightning Bolt"
forge template wording --query "Whenever you sacrifice an artifact"
```

The first shipped sync lane uses Scryfall catalogs for official type and
keyword terms, token-card data for token templates, and Wizards Comprehensive
Rules TXT for the rules wiki, predefined tokens, and counter references.
`mtg-oracle` can inform research, but it is not an app dependency.

## MCP layer

An MTG MCP server can be useful for Codex/AI-assisted design, but it should be optional.

Recommended split:

- app core uses deterministic local caches,
- Codex/MCP uses reference data for suggestions,
- rendering/export never depends on MCP availability.

## Rules-lint examples

- “This card uses deathtouch; typical colors are black/green.”
- “This card is an Aura but has no Enchant line.”
- “This Equipment has no Equip/Reconfigure/For Mirrodin-style attachment rule.”
- “This reskin changed the mana cost but still claims source-card equivalence.”
